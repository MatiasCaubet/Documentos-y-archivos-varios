Categoría Bad Smell Refactoring(s) Qué hace el refactoring
Bloater Long Method Un método que creció tanto que es difícil de entender. Hace demasiadas cosas a la vez. →
Extract Method Extract Parameter
Extract Method: toma un fragmento del método largo y lo convierte en un método separado con nombre descriptivo. Repetir hasta que cada método haga una sola cosa.
Extract Parameter: cuando el método tiene lógica dependiente de parámetros innecesariamente complejos, los extrae.
Bloater Large Class Una clase con demasiados campos, métodos y responsabilidades. Viola el principio de responsabilidad única. →
Extract Class Extract Subclass Extract Method
Extract Class: mueve un grupo de campos y métodos relacionados a una nueva clase separada.
Extract Subclass: cuando algunos comportamientos solo aplican a algunos casos, los mueve a una subclase.
Bloater Long Parameter List Un método recibe demasiados parámetros (más de 3-4). Difícil de entender y llamar correctamente. →
Extract Parameter Introduce Parameter Object
Introduce Parameter Object: agrupa los parámetros relacionados en un objeto que los encapsula, reemplazando la lista larga por un único argumento con significado propio.
Bloater Primitive Obsession Uso de tipos primitivos (String, int) donde debería haber objetos de dominio (ej: "precio" como double en vez de clase Dinero). →
Replace Data Value with Object Introduce Null Object
Replace Data Value with Object: convierte el valor primitivo en un objeto de primera clase con sus propias responsabilidades y comportamiento.
Introduce Null Object: cuando el primitivo es null, introduce un objeto que representa la ausencia de valor.
Bloater Data Clumps El mismo grupo de datos aparece junto en múltiples clases o parámetros (ej: ciudad, país, código postal siempre juntos). Son un objeto que no se creó. →
Extract Class Introduce Parameter Object
Extract Class: crea una clase que encapsula el grupo de datos (ej: clase Direccion).
Introduce Parameter Object: cuando el clump aparece en parámetros de métodos, lo agrupa en un objeto.
Bloater Imperative Loops Bucles for/while con acumuladores y lógica de negocio mezclada. El propósito del loop no es inmediatamente obvio. →
Replace Loop with Pipeline Extract Method
Replace Loop with Pipeline: reemplaza el bucle imperativo con operaciones de colección encadenadas (filter, map, reduce) que expresan la intención con mayor claridad.
Tool Abuser Switch Statements Largos switch/if-else que ramifican según el tipo de un objeto. El mismo switch suele repetirse en varios lugares del código. →
Replace Conditional with Polymorphism Replace State-Altering Conditionals with State
Replace Conditional with Polymorphism: crea una jerarquía de clases donde cada subclase implementa el comportamiento de una rama del switch. El polimorfismo reemplaza la condición.
Replace State-Altering Conditionals with State: cuando el switch controla el estado de un objeto, aplica el patrón State.
Tool Abuser Refused Bequest Una subclase hereda métodos de la superclase pero no los necesita o los ignora. La herencia se usó solo para reutilizar código, no porque exista una relación "es-un". →
Push Down Method Replace Inheritance with Delegation
Push Down Method: baja el comportamiento rechazado a la subclase que sí lo necesita, fuera de la jerarquía compartida.
Replace Inheritance with Delegation: si la subclase realmente no es un tipo de la superclase, reemplaza la herencia por composición.
Tool Abuser Temporary Field Un campo de instancia que solo tiene valor en ciertas circunstancias. El resto del tiempo está vacío/null, lo cual confunde a quien lee el código. →
Extract Class Introduce Null Object
Extract Class: mueve el campo temporal y los métodos que lo usan a una clase separada que solo existe cuando corresponde.
Introduce Null Object: cuando el campo es null en ciertos casos, introduce un objeto que maneja ese caso vacío.
Tool Abuser Alternative Classes with Different Interfaces Dos clases hacen lo mismo pero tienen métodos con nombres o firmas distintas. No son intercambiables aunque deberían serlo. →
Rename Method Move Method Adapter (patrón)
Rename Method: unifica los nombres para que ambas clases hablen el mismo idioma.
Adapter: cuando las interfaces no pueden unificarse fácilmente, introduce un Adapter que hace de puente.
Change Preventer Divergent Change Una clase se modifica por razones diferentes y no relacionadas. Si cada nuevo requerimiento te obliga a cambiar la misma clase por distintos motivos, tiene demasiadas responsabilidades. →
Extract Class
Extract Class: separa las responsabilidades en clases distintas. Cada clase debe tener una sola razón para cambiar (SRP). Divergent Change y Shotgun Surgery son opuestos: en DC una clase cambia por muchas razones; en SS muchas clases cambian por una razón.
Change Preventer Shotgun Surgery Un solo cambio lógico requiere modificar muchas clases distintas al mismo tiempo. La responsabilidad está dispersa donde debería estar unificada. →
Move Method Move Field Inline Class
Move Method / Move Field: consolida el comportamiento disperso en una sola clase que centralice esa responsabilidad.
Inline Class: si una clase quedó tan pequeña que no justifica su existencia, la absorbe la que la llama.
Change Preventer Parallel Inheritance Hierarchies Cada vez que creás una subclase en una jerarquía, tenés que crear una clase correspondiente en otra jerarquía. Caso especial de Shotgun Surgery en herencia. →
Move Method Move Field Form Template Method
Move Method / Move Field: elimina la duplicación moviendo responsabilidades de una jerarquía a la otra hasta que una de las dos desaparece.
Form Template Method: cuando las jerarquías comparten un algoritmo con variaciones, unifica el esqueleto en la superclase.
Dispensable Duplicate Code El mismo fragmento de código aparece en más de un lugar. Si hay que cambiar la lógica, hay que cambiarla en todos los lugares (y siempre se olvida alguno). →
Extract Method Pull Up Method Form Template Method
Extract Method: cuando la duplicación está dentro de la misma clase.
Pull Up Method: cuando la duplicación está en subclases hermanas — sube el método a la superclase.
Form Template Method: cuando los métodos duplicados tienen el mismo esqueleto pero pasos distintos — aplica Template Method para unificar la estructura.
Dispensable Data Class Una clase con solo campos y getters/setters, sin comportamiento propio. Es un contenedor de datos pasivo; la lógica que debería estar aquí vive en otras clases. →
Encapsulate Field Move Method Pull Up Field
Encapsulate Field: protege los campos con acceso controlado (privados con getters/setters), primer paso para darle responsabilidades.
Move Method: mueve a la Data Class los métodos de otras clases que trabajan sobre sus datos — donde viven los datos, debe vivir el comportamiento.
Dispensable Lazy Class Una clase que no hace suficiente para justificar su existencia. Fue creada con buenas intenciones pero quedó pequeña o irrelevante. →
Inline Class Collapse Hierarchy
Inline Class: fusiona la Lazy Class en la clase que la usa, eliminando la abstracción innecesaria.
Collapse Hierarchy: si la Lazy Class es una subclase sin diferencia real con la superclase, las combina.
Dispensable Speculative Generality Código escrito "por si acaso lo necesitamos en el futuro" que hoy nadie usa. Clases abstractas, hooks o parámetros que no tienen usuario real todavía. →
Collapse Hierarchy Inline Class Remove Parameter
Collapse Hierarchy / Inline Class: elimina las abstracciones que no tienen uso real. En TDD esto se evita naturalmente porque solo se escribe código para pasar una prueba existente — nunca código para futuros hipotéticos.
Dispensable Dead Code Variables, parámetros, métodos o clases que ya no se usan en ningún lugar. El código sigue ahí "por las dudas". →
Delete Dead Code
Delete Dead Code: simplemente se elimina. El control de versiones guarda el historial si alguna vez se necesita recuperarlo. No hay ninguna razón para mantener código que nadie llama.
Dispensable Uncommunicative Name Nombres de variables, métodos o clases que no revelan su intención (ej: d, calcular(), Util). El código no se explica solo. →
Rename Method Rename Field
Rename Method / Rename Field: cambia el nombre por uno que revele la intención. Es el refactoring más simple y de mayor impacto en legibilidad. Un buen nombre elimina la necesidad de un comentario.
Dispensable Comments Comentarios que explican qué hace el código (no el porqué). Si el código necesita comentario para entenderse, el código está mal escrito. →
Extract Method Rename Method
Extract Method: extrae el bloque comentado en un método cuyo nombre exprese lo que hacía el comentario. El método bien nombrado reemplaza al comentario.
Los comentarios que explican el porqué (decisiones de diseño no obvias) sí son válidos y no son un smell.
Dispensable Magic Numbers Literales numéricos (o de string) en el código cuyo significado no es obvio: if (status == 3), precio * 1.21. ¿Qué es 3? ¿Qué es 1.21? →
Replace Magic Number with Symbolic Constant
Replace Magic Number with Symbolic Constant: define una constante con nombre significativo (IVA = 1.21, ESTADO_APROBADO = 3) y reemplaza todas las ocurrencias del literal.
Coupler Feature Envy Un método que parece más interesado en los datos de otra clase que en los de la suya propia. Accede constantemente a campos/métodos de otro objeto en lugar de pedirle que haga el trabajo. →
Move Method Extract Method
Move Method: mueve el método a la clase cuyos datos usa — el comportamiento debe vivir donde están los datos.
Extract Method: si solo una parte del método tiene envidia, la extrae y la mueve.
Coupler Message Chains Código del estilo a.getB().getC().getD().hacerAlgo(). El cliente conoce toda la estructura interna de una cadena de objetos. Fragilidad: si cambia un eslabón, todo rompe. →
Hide Delegate Extract Method
Hide Delegate: introduce un método en el objeto inicial que oculta la cadena. El cliente llama a.hacerAlgo() y a internamente navega la cadena. Aplica la Ley de Demeter.
Coupler Middle Man Una clase cuya mayoría de métodos simplemente delegan a otra clase sin agregar valor. Se creó para reducir acoplamiento pero quedó vacía de responsabilidad propia. →
Remove Middle Man Inline Method
Remove Middle Man: elimina la clase intermediaria y hace que el cliente llame directamente al objeto real.
Inline Method: si el método de delegación es trivial, lo inlinea directamente donde se usa.
Coupler Inappropriate Intimacy Dos clases que saben demasiado una de la otra: acceden a campos privados, métodos internos o se conocen mutuamente en exceso. Alto acoplamiento bidireccional. →
Move Method Move Field Extract Class
Move Method / Move Field: redistribuye las responsabilidades para que cada clase solo conozca lo que le corresponde.
Extract Class: si la intimidad refleja que ambas clases comparten una responsabilidad, la extrae en una tercera clase.

📖 Glosario de Refactorings

▼

Extract Method

Toma un fragmento de código y lo convierte en un método separado con nombre que expresa su propósito.

Pull Up Method

Sube un método de subclases hermanas a la superclase cuando ambas tienen la misma implementación.

Pull Up Field

Sube un campo que aparece en múltiples subclases a la superclase común.

Move Method

Mueve un método a la clase cuyos datos usa más frecuentemente.

Move Field

Mueve un campo a la clase que lo usa con mayor frecuencia.

Encapsulate Field

Hace privado un campo público y provee getter/setter, tomando control sobre el acceso.

Rename Method / Field

Cambia el nombre por uno que revele la intención. El refactoring más simple y más impactante.

Form Template Method

Cuando dos métodos en subclases hacen pasos similares en el mismo orden, sube el esqueleto a la superclase como Template Method.

Replace State-Altering Conditionals with State

Reemplaza la lógica condicional que modifica el estado del objeto con el patrón State.

Replace Conditional with Polymorphism

Crea una jerarquía de clases donde cada subclase implementa el comportamiento de una rama del condicional.

Introduce Null Object

Reemplaza los chequeos de null con un objeto inerte que implementa la interfaz esperada sin hacer nada (o con comportamiento por defecto).

Extract Class

Toma parte de los campos y métodos de una clase grande y los mueve a una nueva clase con responsabilidad propia.

Inline Class

Hace lo contrario de Extract Class: fusiona una clase pequeña que no justifica su existencia en la que la usa.

Extract Parameter

Extrae lógica relacionada a parámetros de un método largo en un objeto parámetro separado.

Hide Delegate

Introduce un método que oculta una cadena de mensajes, cumpliendo la Ley de Demeter.

Introduce Local Extension

Crea una subclase o wrapper de una clase que no podés modificar (ej: de una librería externa) para agregar los métodos que necesitás.

Replace Magic Number with Symbolic Constant

Define una constante con nombre significativo y reemplaza todos los literales "mágicos".
